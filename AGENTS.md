# AGENTS.md

이 문서는 프로젝트에서 작업하는 모든 에이전트가 가장 먼저 읽어야 하는 최상위 지침이자 문서 탐색 지도다.

## 핵심 원칙

### 1. 문서화는 모든 작업의 일부다

- 프로젝트의 중요한 정보, 장기적으로 지켜야 할 규칙, 반복해서 필요한 지식은 반드시 문서로 남긴다.
- 최상위 지침과 문서 탐색 정보는 이 문서에 유지한다.
- 구현 세부 사항은 `docs/` 아래의 적절한 문서에 기록하고, 이 문서의 문서 지도에서 해당 문서로 안내한다.
- 코드, 구조, 동작, 의사결정이 바뀌면 관련 문서의 갱신 필요성을 반드시 확인한다.
- 작업 완료 전 코드와 문서가 서로 일치하는지 검토한다. 필요한 문서 갱신까지 끝나야 작업이 완료된 것으로 본다.
- 새 문서를 추가하거나 기존 문서의 역할을 바꾸면 아래 문서 지도와 `docs/README.md`를 함께 갱신한다.
- 시간이 지나 구현 배경을 잊더라도 새로운 에이전트가 문서만으로 필요한 정보를 찾아 작업할 수 있어야 한다.

### 2. 구현 전에 선택지를 제시하고 합의한다

- 사용자가 기능이나 변경을 요청하면 구현부터 시작하지 않는다.
- 먼저 가능한 접근 방법을 제시하고 각 방법의 장점, 단점, 트레이드오프를 설명한다.
- 추천안을 명확히 밝히고 그 이유를 설명한다.
- 추천하거나 선택지를 구성하는 데 필요한 정보가 부족하면 먼저 사용자에게 질문한다.
- 사용자가 접근 방법을 선택하거나 명시적으로 진행에 동의한 뒤에 구현한다.
- 이미 사용자가 구체적인 접근 방법을 선택해 지시한 경우에는 같은 선택을 다시 묻지 않아도 된다. 다만 중대한 부작용이나 더 나은 대안이 발견되면 구현 전에 알린다.

### 3. 불확실하면 반드시 질문한다

- 요구사항, 용어, 범위, 의도 또는 성공 기준이 명확하지 않으면 추측해서 구현하지 않는다.
- 서로 다르게 해석할 수 있는 부분을 구체적으로 설명하고 사용자에게 확인한다.
- 질문을 줄이기 위해 임의의 결정을 내리지 않는다. 잘못된 구현보다 명확한 질문을 우선한다.

### 4. 테스트는 구현과 함께 유지한다

- 새 기능, 동작 변경, 버그 수정에는 해당 동작을 검증하는 테스트를 함께 추가하거나 수정한다.
- 코드 구조나 요구사항이 바뀌면 기존 테스트가 새 동작과 경계를 올바르게 검증하는지 반드시 확인한다.
- 더 이상 유효하지 않은 테스트를 방치하지 않는다. 변경된 요구사항에 맞게 수정하고, 검증 대상 자체가 사라졌을 때만 삭제한다.
- 가능한 경우 버그를 수정하기 전에 실패를 재현하는 테스트를 먼저 추가한다.
- 외부 API를 사용하는 코드는 순수 로직과 연동 경계를 분리하고, 실제 외부 서비스 없이도 핵심 동작과 오류 처리를 반복 검증할 수 있게 한다.
- 변경 범위에 맞는 테스트를 실행하고 결과를 확인한다. 필요한 테스트의 추가·갱신과 검증까지 끝나야 구현이 완료된 것으로 본다.
- 테스트할 수 없는 변경이거나 테스트를 추가하지 않는 편이 타당하다면 그 이유와 대체 검증 방법을 사용자에게 명확히 알린다.

## 기본 작업 순서

1. 이 문서와 현재 작업에 해당하는 세부 문서를 읽는다.
2. 요구사항의 불확실한 부분을 확인하고 필요하면 질문한다.
3. 가능한 접근 방법, 트레이드오프, 추천안을 사용자에게 제시한다.
4. 사용자의 선택 또는 동의를 받은 뒤 구현한다.
5. 변경할 동작의 성공 기준을 정하고 필요한 테스트를 추가하거나 갱신한다.
6. 구현한 뒤 변경 범위에 맞는 테스트와 기타 검증을 실행한다.
7. 관련 문서를 갱신하고 문서, 테스트, 구현의 정합성을 확인한다.

## 문서 지도

```text
AGENTS.md                         # 최상위 원칙과 문서 탐색 지도
└── docs/
    ├── README.md                 # 상세 문서 목록, 작성·관리 규칙, 새 문서 추가 기준
    ├── product-overview.md       # 제품 목표, 범위, 확정 사항과 미결정 사항
    ├── brand.md                  # 현재 제품명, 브랜드 콘셉트와 말투 원칙
    ├── architecture/
    │   └── runtime.md            # 기술 스택, 실행 방식과 컴포넌트 경계
    ├── data/
    │   └── google-sheets.md      # 공동 편집 Google Sheet의 스키마와 관리 규칙
    ├── features/
    │   ├── candidate-management-alpha.md # 별도 DB와 후보 추가·편집 도구의 알파 범위와 전환 기준
    │   ├── daily-recommendation.md # 정기 추천의 메시지와 예외 처리 정책
    │   ├── leaderboard.md        # 사내 리더보드의 지표, 화면과 캐시·접근 정책
    │   └── reactions.md          # 후보별 번호 반응과 통계 데이터 수집 규칙
    ├── integrations/
    │   └── slack.md              # Slack Manifest, 권한, 설치·채널 연결과 인계
    └── operations/
        ├── development.md        # 현재 구현 상태, 코드 구조, 개발·검증 및 배포 템플릿 주의
        ├── google-access.md      # Google 자원의 조직 소유, 권한과 담당자 인계
        └── sustainability.md     # 장기 운영 규모 검증과 운영상 주의 사항
```

### 상황별로 읽을 문서

- 어떤 작업이든 시작할 때: `AGENTS.md`
- 상세 문서를 찾거나 새 문서를 추가할 때: `docs/README.md`
- 제품의 목표, 현재 범위, 단계 구분을 확인할 때: `docs/product-overview.md`
- 제품명, 브랜드 콘셉트, 사용자에게 보이는 문구를 작성할 때: `docs/brand.md`
- 기술 스택, 프로세스 형태, 스케줄러와 외부 연동 경계를 확인할 때: `docs/architecture/runtime.md`
- Google Sheet의 열, 값 형식, 공동 편집 규칙을 확인할 때: `docs/data/google-sheets.md`
- 정기 추천 시각, 메시지 구성, 후보 부족과 운영 오류의 처리 방식을 확인할 때: `docs/features/daily-recommendation.md`
- 후보 관리 알파의 격리 범위, SQLite 구조, 백업·복원과 시트 복귀 기준을 확인할 때: `docs/features/candidate-management-alpha.md`
- 사내 리더보드의 표시 지표, 정렬, 갱신 주기와 접근 범위를 확인할 때: `docs/features/leaderboard.md`
- 후보별 좋아요, Slack 번호 반응, 반응 집계와 통계 의미를 확인할 때: `docs/features/reactions.md`
- Slack 앱을 생성·설치하거나 권한, 채널, 토큰 또는 담당자 인계를 확인할 때: `docs/integrations/slack.md`
- 현재 구현 범위, 파일별 책임, 개발 환경, 테스트 또는 systemd 템플릿을 확인할 때: `docs/operations/development.md`
- Google Sheet, Cloud 프로젝트, Service Account의 소유권이나 담당자 인계를 확인할 때: `docs/operations/google-access.md`
- 용량, API 한도, 장기 운영 위험과 확장 시점을 확인할 때: `docs/operations/sustainability.md`
- 필요한 세부 문서가 아직 없다면: `docs/README.md`의 기준에 따라 문서를 만들고 이 지도를 갱신한다.
