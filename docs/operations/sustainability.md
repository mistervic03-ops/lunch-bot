# 장기 운영 지속 가능성

## 검증 범위

밥라투스트라를 Slack 채널 하나에서 월요일부터 금요일까지 하루 한 번 실행하고, 정상 추천마다 최대 세 개의 추천 로그를 기록하는 현재 범위를 기준으로 검증한다.

검증일은 2026년 7월 20일이다. 외부 서비스의 한도와 정책은 바뀔 수 있으므로 기술 스택을 확정하거나 관련 장애가 발생하면 공식 문서를 다시 확인한다.

## 연간 데이터 규모

공휴일을 제외하지 않아도 1년의 평일은 최대 약 262일이다.

```text
최대 실행 횟수: 약 262회/년
추천 로그: 262 × 3 = 최대 786행/년
로그 셀: 786 × 9 = 최대 7,074셀/년
```

10년을 운영해도 추천 로그는 약 7,860행과 70,740셀 수준이다. `lunch_options`의 예상 규모까지 더해도 Google Sheets의 파일당 1,000만 셀 한도에 비해 매우 작다.

따라서 용량만을 이유로 현재 운영 원본에 다음 기능이 필요하지 않다.

- 서버형 관계형 데이터베이스
- 추천 로그 자동 삭제 또는 연도별 보관
- 영속 로그 캐시나 검색 인덱스
- 데이터 압축 또는 집계 작업

## API 사용량

정상 실행은 Google Sheet의 후보와 로그를 읽고, 최근 다섯 개의 Slack 메시지 반응을 조회하고, 좋아요를 요청 한 번으로 갱신한 뒤 새 메시지 하나와 번호 반응 최대 세 개를 게시하고 로그를 요청 한 번으로 추가하는 정도다.

웹 리더보드는 접속 요청마다 API를 호출하지 않고 프로세스 메모리에 5분간 스냅샷을 유지한다. 캐시가 만료된 뒤 첫 요청만 후보와 로그를 각각 한 번 읽는다. 하루 종일 지속해서 접속해도 최대 5분당 두 번의 읽기이며, 실제 사내 사용량에서는 이보다 훨씬 적다. 서버 재시작 때 캐시는 사라지며 원본 데이터는 항상 Google Sheet다.

Google Sheets API의 기본 한도는 프로젝트당 분당 읽기 300회와 쓰기 300회이며, 사용자당 프로젝트 기준으로 각각 분당 60회다. Slack은 일반적으로 채널당 초당 메시지 하나를 허용한다. 하루 한 번 실행하는 밥라투스트라의 사용량은 이 한도보다 매우 작다.

- [Google Sheets API 사용 한도](https://developers.google.com/workspace/sheets/api/limits)
- [Google Sheets 파일 크기 한도](https://support.google.com/drive/answer/37603)
- [Slack Web API 사용 한도](https://docs.slack.dev/apis/web-api/rate-limits/)

## 장기 운영에서 더 중요한 위험

용량보다 계정과 권한 관리가 더 중요하다.

- Google Cloud 프로젝트와 Slack 앱을 개인 계정 한 명에게만 의존하지 않는다.
- Google Sheet와 배포 환경에 최소 두 명의 사내 운영 담당자가 접근할 수 있게 한다.
- Slack 토큰과 Google 인증 정보는 저장소에 커밋하지 않고 배포 환경의 비밀 저장 기능을 사용한다.
- `recommendation_log` 탭은 봇만 수정하고 직원은 읽기 전용으로 보호한다.
- Google Sheet 탭 이름과 열 구조를 바꿀 때는 코드와 문서를 같은 작업에서 갱신한다.
- 운영 오류는 비공개 `#밥괘-운영` 채널에서 확인한다.

## 의도적으로 허용하는 한계

- 식당과 메뉴의 안정적인 내부 ID를 두지 않는다.
- 식당명이 수정되면 과거 로그와 연결되지 않아 새 식당처럼 우선 추천될 수 있다.
- 이름 변경은 드물고 영향도 일시적인 추천 순서 변화에 그치므로, 초기 복잡성을 늘리지 않고 이 한계를 허용한다.
- 식당명 연결 문제가 실제로 반복될 때만 내부 ID 도입을 검토한다.

## 구조를 다시 검토할 조건

다음 요구가 생기면 Google Sheets만으로 충분한지 다시 평가한다.

- 여러 Slack 워크스페이스나 다수의 추천 채널을 지원함
- 개인별 선호와 추천 이력을 저장함
- 투표, 통계 또는 복잡한 검색을 실시간으로 제공함
- Google Sheet 편집 충돌이나 성능 문제가 실제로 발생함
- 식당과 메뉴의 변경 이력을 정확히 추적해야 함

후보 입력 편의와 변경 이력의 효용은 용량과 별개의 제품 가설이므로 작은 SQLite 관리 도구를 병행 알파로 검증한다. 이는 이미 DB가 필요하다고 확정한 것이 아니며, 전환·복귀 기준과 백업 조건은 [`../features/candidate-management-alpha.md`](../features/candidate-management-alpha.md)를 따른다.
