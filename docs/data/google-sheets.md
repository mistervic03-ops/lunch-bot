# Google Sheet 데이터 명세

## 역할

Google Sheet는 초기 점심 추천 후보의 원본이자 구성원이 함께 사용하는 관리 화면이다. 별도의 데이터베이스나 관리 대시보드는 초기 범위에 포함하지 않는다.

한 행은 하나의 `식당 + 메뉴` 추천 후보를 의미한다. 같은 식당에 추천 메뉴가 여러 개라면 메뉴마다 행을 하나씩 추가한다.

## 시트 이름

추천 후보와 추천 이력을 각각 별도 탭에 보관한다.

```text
lunch_options          # 직원 공동 편집
recommendation_log     # 봇 기록, 직원 읽기 전용
인기 메뉴               # 수식 기반 누적 좋아요 리더보드, 직원 읽기 전용
```

설정 탭은 필요성이 확인되기 전에는 추가하지 않는다.

현재 실제 Spreadsheet는 [밥라투스트라 점심 추천](https://docs.google.com/spreadsheets/d/1pyOGBbDwAZurNSTY0foQmTg7mojEGhYD3DzMal-WeSc/edit)이며, 두 탭과 고정 헤더가 생성되어 있다. 최초 생성 때는 예시 데이터를 넣지 않았고 이후 실제 후보를 등록해 사용한다. 후보 수는 계속 바뀌므로 문서의 고정값으로 취급하지 않는다.

## 열 명세

직원이 보는 열 이름과 순서는 다음과 같이 고정한다. 개발용 변수명을 Sheet에 노출하지 않는다.

| 순서 | 열 이름 | 필수 | 형식 | 설명 |
|---:|---|:---:|---|---|
| 1 | `사용` | 예 | 체크박스 | 체크된 행만 추천 대상으로 사용한다. |
| 2 | `식당` | 예 | 문자열 | 식당 이름이다. |
| 3 | `추천 메뉴` | 예 | 문자열 | 추천 메뉴 이름이다. |
| 4 | `가격(원)` | 아니요 | 0 이상의 정수 | 1인 기준 가격을 원 단위 숫자로 입력한다. |
| 5 | `지도 링크` | 아니요 | URL | 네이버지도, 카카오맵 등 지도 서비스의 링크이다. |
| 6 | `추천인` | 아니요 | 문자열 | 추천한 구성원의 이름 또는 Slack 표시명이다. |
| 7 | `한줄 메모` | 아니요 | 문자열 | 양, 맵기, 혼잡도 등 짧은 참고 정보이다. |

예시는 다음과 같다.

| 사용 | 식당 | 추천 메뉴 | 가격(원) | 지도 링크 | 추천인 | 한줄 메모 |
|:---:|---|---|---:|---|---|---|
| TRUE | 마루식당 | 제육볶음 | 10000 |  | 철수 | 양이 많음 |
| TRUE | 마루식당 | 김치찌개 | 9000 |  | 영희 |  |
| TRUE | 소바집 | 냉소바 | 11000 | https://example.com/map | 민지 | 여름 추천 |

## 공동 관리 규칙

- 모든 구성원은 승인 없이 행을 추가하고 값을 수정할 수 있다.
- 추천 대상에서 제외할 때는 행을 바로 삭제하기보다 `사용` 체크를 해제한다.
- 첫 번째 헤더 행과 열 이름 및 순서는 변경하지 않는다.
- `식당` 또는 `추천 메뉴`가 비어 있는 행은 추천 대상에서 제외한다.
- `가격(원)`은 통화 기호나 쉼표 없이 숫자로 입력한다. Sheet와 Slack에서 통화 형식으로 표시한다.
- `지도 링크`는 선택값이다. 비어 있으면 링크 없이 추천하며, 값이 있으면 Slack 메시지에 표시한다.
- 같은 식당과 메뉴 조합이 이미 있는지 확인한 뒤 새 행을 추가한다.
- 지도 링크는 특정 서비스로 제한하지 않는다.

## 읽기와 입력 오류 처리

- 봇은 `lunch_options!A:G`를 한 번에 읽고 계산 중에는 Sheet를 수정하지 않는다.
- 값은 숫자와 체크박스를 정확히 판별하기 위해 Sheets API의 `UNFORMATTED_VALUE` 형식으로 읽는다.
- 헤더 일곱 개의 이름과 순서가 명세와 정확히 다르면 전체 실행을 중단한다.
- 완전히 빈 행과 `사용` 체크가 해제된 행은 조용히 제외한다.
- 활성 행의 `사용`, 필수값, 가격 또는 지도 URL이 잘못되면 해당 행만 제외하고 행 번호와 이유를 운영 흐름에 반환한다. 다른 유효한 행은 계속 사용한다.
- `사용`은 Sheet가 제공하는 기본 체크박스만 사용한다.
- `가격(원)`은 0 이상의 정수만 허용한다.
- `지도 링크`는 비어 있거나 `http://` 또는 `https://` URL이어야 한다.
- 중복된 식당과 메뉴 행은 이번 검증 단계에서 입력 오류로 취급하지 않는다. 추천 계산이 같은 조합을 한 번만 사용한다. 별도의 경고나 수정 정책은 필요성이 확인되면 결정한다.

## 직원용 서식과 보호

- 첫 번째 헤더 행은 제품의 보라색 계열로 표시하고 화면 상단에 고정한다.
- 각 헤더 셀의 메모에서 입력 방법과 선택 여부를 확인할 수 있다.
- `사용` 열의 데이터 행에는 체크박스를 제공한다.
- `가격(원)`은 숫자로 저장하면서 화면에는 천 단위 구분과 `원`을 표시한다.
- 입력 내용을 읽기 쉽도록 열 너비와 줄바꿈을 설정한다.
- 내용이 있는 행에서 `사용` 체크를 해제하면 행 전체를 회색으로 표시한다.
- 후보 시트의 헤더 행은 소유자만 수정할 수 있게 보호한다.
- `recommendation_log` 전체 탭은 소유자와 Service Account만 수정할 수 있게 보호한다.

코드에서는 열 순서에 따라 각각 `active`, `restaurant`, `menu`, `price`, `map_url`, `recommended_by`, `note` 내부 필드로 매핑한다. 이 이름은 구현 내부에서만 사용하며 Sheet에는 표시하지 않는다.

## `recommendation_log` 열 명세

한 번의 정기 추천에서 선택된 식당과 메뉴를 각각 한 행으로 기록한다. 정상적으로 식당 세 곳을 게시하면 로그 세 행이 추가된다.

| 순서 | 열 이름 | 형식 | 설명 |
|---:|---|---|---|
| 1 | `추천 시각` | KST 날짜와 시각 | 실제 Slack 게시가 성공한 시각이다. |
| 2 | `추천 날짜` | `YYYY-MM-DD` | 같은 날짜의 중복 실행을 확인하는 기준이다. |
| 3 | `순서` | `1`, `2`, `3` | 메시지 안의 후보 순서이자 번호 반응과 연결하는 값이다. |
| 4 | `식당` | 문자열 | 게시 당시의 식당명이다. |
| 5 | `메뉴` | 문자열 | 게시 당시의 메뉴명이다. |
| 6 | `Slack 채널 ID` | 문자열 | 추천을 게시한 Slack 채널 ID이다. |
| 7 | `Slack 메시지 ID` | 문자열 | 게시된 Slack 메시지 식별자이다. |
| 8 | `좋아요 수` | 0 이상의 정수 | 해당 번호 반응 수에서 봇 자신의 반응을 제외한 값이다. |
| 9 | `좋아요 집계 시각` | KST 날짜와 시각 | Slack에서 반응 수를 마지막으로 동기화한 시각이다. |

- 봇만 행을 추가하고 직원은 읽기 전용으로 사용한다.
- Slack 메시지 게시에 성공한 추천만 기록한다.
- 식당이나 메뉴 이름이 나중에 수정되더라도 당시 게시 내용을 보존하기 위해 이름을 스냅샷으로 기록한다.
- 로그는 장기간 운영해도 규모가 작으므로 자동 삭제하거나 별도 보관하지 않는다.
- 안정적인 내부 식별자는 초기 범위에 추가하지 않는다. 식당명이 바뀌면 추천 알고리즘에서 새 식당처럼 취급될 수 있다는 한계를 허용한다.
- `좋아요 수`는 개인별 투표 이력이 아니라 Slack에 표시된 전체 반응 수의 스냅샷이다.
- 최근 다섯 개의 밥라투스트라 메시지를 매일 다시 확인하므로 해당 기간 안의 늦은 반응은 다음 동기화에서 반영된다.

## 추천 로그 읽기와 쓰기 규칙

- 봇은 `recommendation_log!A:I` 전체를 `UNFORMATTED_VALUE`로 읽는다.
- 보호된 봇 전용 탭이므로 헤더나 데이터 행 하나라도 명세와 다르면 행을 건너뛰지 않고 전체 실행을 중단한다. 잘못된 로그를 무시하면 공정 순환과 중복 게시 방지가 함께 깨질 수 있기 때문이다.
- `추천 시각`과 `좋아요 집계 시각`은 `YYYY-MM-DD HH:MM:SS` 형식의 KST 값으로 기록한다.
- `추천 날짜`는 `YYYY-MM-DD`, `순서`는 `1`부터 `3`, `좋아요 수`는 0 이상의 정수만 허용한다.
- `추천 시각`의 KST 날짜와 `추천 날짜`가 일치해야 한다.
- Slack 게시가 성공한 뒤 해당 메시지의 후보 행을 Sheets API 요청 한 번으로 추가한다.
- 일부 행만 기록된 것으로 응답되면 로그 쓰기 실패로 처리한다.
- 좋아요 동기화는 로그를 읽을 때 확보한 실제 Sheet 행 번호를 사용해 `좋아요 수`와 `좋아요 집계 시각`만 일괄 갱신한다.
- 일부 좋아요 행만 갱신된 것으로 응답되면 동기화 실패로 처리하고 당일 추천은 계속한다.

## 추천 로직에서 지켜야 할 데이터 해석

- `사용`이 체크되고 `식당`과 `추천 메뉴`가 모두 있는 행만 유효한 후보다.
- 같은 식당에 여러 메뉴 행이 있을 수 있다.
- 한 번의 추천에서는 식당 이름을 기준으로 중복을 제거하여 서로 다른 식당 세 곳을 선택한다.
- 선택값이 없다는 이유로 유효한 행 전체를 제외하지 않는다.

식당과 메뉴를 선택하는 자세한 규칙은 [`../features/daily-recommendation.md`](../features/daily-recommendation.md)를 따른다.

## `인기 메뉴` 리더보드

`인기 메뉴` 탭은 `recommendation_log`의 전체 기록을 `식당 + 메뉴`로 묶어 보여주는 읽기 전용 리더보드다. 현재 `lunch_options`에서 비활성화되거나 삭제된 메뉴도 과거 기록을 보존하기 위해 포함한다.

| 순서 | 열 이름 | 설명 |
|---:|---|---|
| 1 | `순위` | 현재 정렬 결과의 1부터 시작하는 순위다. |
| 2 | `식당` | 추천 당시 기록된 식당명이다. |
| 3 | `메뉴` | 추천 당시 기록된 메뉴명이다. |
| 4 | `누적 좋아요` | 해당 식당과 메뉴의 `좋아요 수` 합계다. |
| 5 | `추천 등장 횟수` | 해당 식당과 메뉴가 추천 로그에 기록된 횟수다. |

- 누적 좋아요 내림차순으로 정렬하고, 동률이면 식당명과 메뉴명 오름차순으로 순서를 고정한다.
- `B1`의 `QUERY` 수식이 식당, 메뉴, 누적 좋아요와 추천 등장 횟수를 집계한다.
- `A2`의 `ARRAYFORMULA`가 결과 행에 순위를 붙인다.
- 좋아요 집계가 `recommendation_log`에 반영되면 별도 봇 쓰기 없이 자동으로 다시 계산된다.
- 탭 전체는 기존 `recommendation_log`와 같은 소유자와 Service Account만 수정할 수 있게 보호한다.
- 리더보드는 인기 현황을 보여줄 뿐 일일 공정 순환 추천의 입력으로 사용하지 않는다.
- 사내 웹 리더보드는 수식 결과를 다시 읽지 않고 `lunch_options`와 `recommendation_log` 원본을 동일한 의미로 집계한다. 웹 화면이 추가되어도 이 탭은 Sheet 사용자를 위해 유지한다.
- 수식을 복구할 때는 `B1`에 아래 `QUERY` 수식을, `A2`에 아래 `ARRAYFORMULA` 수식을 입력한다. 수식과 전체 탭의 보호 편집자는 `recommendation_log`와 같은 소유자 및 Service Account로 유지한다.

```text
=QUERY('recommendation_log'!D2:H,"select D, E, sum(H), count(E) where D is not null and E is not null group by D, E order by sum(H) desc, D asc, E asc label D '식당', E '메뉴', sum(H) '누적 좋아요', count(E) '추천 등장 횟수'",0)
=ARRAYFORMULA(IF(B2:B="","",ROW(B2:B)-1))
```

Google 자원의 조직 소유와 담당자 인계는 [`../operations/google-access.md`](../operations/google-access.md)를 따른다.
