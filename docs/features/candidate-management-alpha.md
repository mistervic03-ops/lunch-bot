# 후보 관리 도구 알파

## 목적과 현재 상태

Google Sheet보다 입력 장벽은 낮추면서 중복 방지, 수정 이력, 안전한 비활성화를 제공할 수 있는지 실제 화면으로 검증한다. 이 단계는 **병행 알파**다. 정기 추천, Slack 버튼과 전당은 계속 Google Sheet를 원본으로 사용하며 SQLite 알파 DB를 읽거나 쓰지 않는다.

알파가 기대에 못 미치면 코드와 알파 DB만 걷어내면 된다. 기존 Sheet와 운영 서비스에는 데이터 변환이나 복구 작업이 필요 없다.

## 사용자 흐름

- 첫 화면은 `식당`과 `추천 메뉴`만 필수로 받는다.
- 가격, 지도 링크, 추천인과 한줄 메모는 접힌 선택 영역에 둔다.
- 같은 식당과 메뉴 조합은 공백과 대소문자를 정규화해 중복을 막고 기존 후보로 안내한다.
- 후보 목록에서 검색하고 정보를 수정할 수 있다.
- 삭제는 제공하지 않는다. 대신 추천에서 잠시 빼거나 다시 넣는다.
- 최근 수정에서 변경 주체와 종류를 읽기 전용으로 확인한다.
- 브라우저는 추천인 이름만 해당 기기의 `localStorage`에 기억한다. 서버의 권한이나 데이터 근거로 사용하지 않는다.

로그인은 알파 범위에서 제외한다. 사내망 접근 제한과 직원 간 신뢰를 전제로 하되, 브라우저의 쓰기 요청은 같은 origin에서 온 경우에만 받는다. 사내망 밖 공개, 여러 조직 지원 또는 악의적인 변경이 실제 위험이 되면 인증을 먼저 도입한다.

## 데이터와 안전장치

SQLite 파일 하나에 다음을 보관한다.

```text
candidates           # 안정적인 ID, 활성 여부와 후보 정보
recommendation_log   # 전환 가능성 검증을 위해 가져온 과거 추천 기록
candidate_changes    # 추가·수정·비활성화·복귀 이력
```

- WAL 모드, foreign key, 5초 busy timeout을 사용한다.
- systemd timer가 매일 03:30 KST에 SQLite online backup API로 전체 스냅샷을 만든 뒤 integrity check를 실행한다.
- 새 백업이 성공하면 이름 기준 최신 30개만 남기고 밥라투스트라 패턴의 오래된 백업만 삭제한다. 수동으로 이름을 바꾼 파일은 건드리지 않는다.
- 백업 파일은 마이크로초를 포함한 UTC 이름을 사용한다.
- 운영 전환 전에는 백업 디렉터리를 서버 밖에서도 복구 가능한 위치로 복제해야 한다. 알파의 로컬 백업만으로는 서버 자체 손실을 견디지 못한다.
- 스키마 변경이 생기면 명시적인 버전과 복원 테스트를 먼저 추가한다. 지금은 schema version 1이며 migration은 없다.

## Sheet에서 최초 가져오기

`python -m bapratustra --import-sheet-to-alpha`는 Service Account의 읽기 전용 범위로 `lunch_options`와 `recommendation_log`를 한 번 읽는다. 비활성 후보도 보존한다. 후보 행에 검증 오류가 하나라도 있거나 대상 DB가 비어 있지 않으면 아무 것도 가져오지 않고 중단한다. Sheet에는 쓰지 않는다.

```bash
python -m bapratustra --import-sheet-to-alpha
python -m uvicorn bapratustra.alpha_web:create_app --factory --host 127.0.0.1 --port 8031
python -m bapratustra --backup-alpha
```

필요한 별도 환경변수는 `BAPRATUSTRA_ALPHA_DB`와 `BAPRATUSTRA_ALPHA_BACKUP_DIR`다. 알파 웹은 Slack과 Google 설정 없이 실행할 수 있다. 가져오기 명령만 Google 설정을 함께 요구한다.

운영에서는 `bapratustra-alpha.service`가 웹을 상시 실행하고 비정상 종료 때 자동 재시작한다. `bapratustra-alpha-backup.timer`와 `bapratustra-alpha-backup.service`가 일일 백업과 최근 30개 보존을 담당한다. 웹 요청마다 백업하거나 화면에서 변경을 되돌리는 기능은 의도적으로 두지 않는다.

복원은 서비스를 멈춘 상태에서 검증된 백업 파일을 운영 DB 경로로 복사하는 운영 작업이다. 자동 복원 버튼은 실수의 영향이 커서 제공하지 않는다. 실제 서버에서 파일 교체나 서비스 재시작이 필요하면 운영 승인을 받은 뒤 수행한다.

## 평가와 전환 기준

다음 조건을 짧은 사내 사용에서 확인한 뒤에만 원본 전환을 제안한다.

- 휴대폰에서 식당과 메뉴 등록을 설명 없이 끝낼 수 있다.
- 중복 안내, 검색, 수정, 추천 제외가 Sheet보다 명확하다.
- 운영 담당자가 최근 수정과 자동 백업 상태를 확인할 수 있다.
- 실제 후보가 누락되거나 중복되는 사례가 없다.
- 최소 한 명의 후임 운영자가 DB 파일, 백업, 실행 방식과 복구 절차를 인계받는다.

통과하면 별도 작업에서 봇과 전당의 저장소를 SQLite로 전환하고, 전환 직전 Sheet 스냅샷과 DB 백업을 함께 남긴다. 통과하지 못하면 Sheet를 계속 원본으로 두고 알파를 종료한다. 알파 중 새로 들어온 유효 후보는 사람이 검토해 Sheet에 옮긴다. 양방향 동기화는 충돌과 이중 원본을 만들기 때문에 구현하지 않는다.

## 알파에서 의도적으로 제외한 것

- Google 또는 Slack 로그인과 역할별 권한
- 영구 삭제
- Sheet와 SQLite의 실시간 또는 양방향 동기화
- 별도 JavaScript SPA, ORM, PostgreSQL, Redis
- 봇과 전당의 SQLite 전환
- 관리자용 범용 대시보드와 복잡한 통계
- 웹 화면에서의 데이터 복원과 변경 되돌리기
