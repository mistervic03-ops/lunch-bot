# 후보 간편 등록 웹

## 목적과 원칙

구성원이 Google Sheet의 열 구조를 몰라도 식당과 메뉴를 빠르게 보탤 수 있는 사내망 웹 화면을 제공한다. Google Sheet는 계속 유일한 데이터 원본이다. 웹과 Sheet는 서로 다른 입력 화면일 뿐 별도 데이터베이스나 동기화 계층을 두지 않는다.

```text
구성원
├── 간단한 한 건 등록·검색 → 후보 간편 등록 웹
└── 자세한 수정·비활성화·일괄 관리 → Google Sheet
                                      ↓
                              lunch_options 단일 원본
                                      ↓
                              정기 추천과 전당
```

## 사용자 흐름

- 등록 화면은 `식당`과 `추천 메뉴`만 필수로 받는다.
- 가격, 지도 링크, 추천인과 한줄 메모는 접힌 선택 영역에 둔다.
- 제출 전에 Sheet의 활성·비활성 후보를 모두 읽어 같은 식당과 메뉴 조합을 확인한다.
- 공백과 대소문자를 정규화한 뒤 같은 조합이면 추가하지 않고 Sheet 확인 링크를 안내한다.
- 중복 후보가 비활성 상태면 Sheet에서 `사용`을 다시 체크하도록 상태를 표시한다.
- 새 후보는 `사용=TRUE`인 한 행으로 `lunch_options!A:G`에 추가한다.
- 후보 검색과 상태·가격·추천인·지도 링크 조회는 웹에서 제공한다.
- 활성 후보에는 상태 문구를 반복하지 않고 비활성 후보에만 `추천 제외`를 표시한다.
- 수정, 비활성화, 복귀와 여러 행 관리는 `Google Sheet 관리` 링크로 넘긴다.
- 등록 화면의 주 버튼은 `후보 등록`, 검색 버튼은 `검색`, 지도 링크는 `지도 보기`로 표시한다.
- 추천인 이름만 해당 브라우저의 `localStorage`에 기억한다. 권한이나 신원 근거로 사용하지 않는다.

Sheet의 정확한 열과 검증 규칙은 [`../data/google-sheets.md`](../data/google-sheets.md)를 따른다.

## 오류와 동시성

- 브라우저의 쓰기 요청은 같은 origin에서 온 경우에만 받는다. 이는 로그인이나 사용자 인증을 대신하지 않는다.
- Sheet 읽기 또는 추가가 실패하면 기술 상세를 노출하지 않고 잠시 후 재시도하도록 안내한다.
- 후보 추가 성공은 Sheets API가 정확히 한 행을 추가했다고 응답한 경우에만 표시한다.
- 중복 확인과 행 추가는 하나의 원자적 연산이 아니다. 두 사람이 같은 후보를 동시에 제출하면 중복 행이 생길 수 있으나 사내 소규모·저빈도 입력에서는 이 한계를 허용한다.
- 완전한 중복 방지를 위해 별도 DB나 잠금 서비스를 추가하지 않는다. 중복이 실제로 반복되면 Sheet에서 정리하거나 구조를 다시 검토한다.

## 실행과 운영

로컬 확인은 기존 Google 설정으로 실행한다.

```bash
python -m uvicorn bapratustra.alpha_web:create_app --factory --host 127.0.0.1 --port 8031
```

운영에서는 `bapratustra-alpha.service`가 Uvicorn 단일 worker를 상시 실행하고 비정상 종료 때 자동 재시작한다. 별도 DB, migration, 데이터 백업 timer와 추가 환경변수는 없다. Service Account에는 기존 Spreadsheet 편집 권한이 필요하다.

`/healthz`는 Google API를 호출하지 않고 프로세스 상태만 반환한다. 실제 Sheet 연결은 등록·검색 화면에서 확인한다.

## 의도적으로 제외한 것

- 별도 SQLite 또는 PostgreSQL
- Sheet와 다른 데이터 원본
- 양방향 동기화와 주기적 내보내기
- 웹에서의 수정·삭제·비활성화와 일괄 편집
- 로그인과 역할별 권한
- 관리자용 범용 대시보드
