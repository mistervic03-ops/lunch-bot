# 정기 점심 추천

## 목적

밥라투스트라는 점심 전용 옵트인 Slack 채널에 정해진 시간마다 점심 후보를 게시하여, 구성원이 메뉴를 고르고 함께 식사할 사람을 찾는 대화를 시작하도록 돕는다.

## 게시 일정

- 월요일부터 금요일까지 오전 11시(KST)에 게시한다.
- 대한민국 공휴일을 별도로 판별하지 않는다. 공휴일이 평일이면 게시 대상이다.
- 자동 메시지에 `@channel` 또는 `@here` 멘션을 사용하지 않는다.
- 개인 명령으로 자신만의 추천을 받는 기능은 초기 범위에 포함하지 않는다.

## 정상 추천

- 한 번에 서로 다른 식당 세 곳을 추천한다.
- 한 식당에 메뉴 행이 여러 개 있으면 그중 한 메뉴만 표시한다.
- 추천 후보를 식당별 카드로 나누지 않고 하나의 간결한 목록으로 표시한다.
- 브랜드 이모지는 남발하지 않는다. 제목의 `📜` 하나만 사용한다.
- 등장과 선언은 캐릭터를 강하게 표현하고, 후보 정보와 반응 안내는 간결하고 친절하게 쓴다.
- em dash는 후보 행의 식당과 메뉴 구분 외에는 사용하지 않는다.
- 메시지 게시 후 후보 순서에 맞는 `1️⃣`, `2️⃣`, `3️⃣` 반응을 봇이 추가한다. 번호 반응은 장식이 아니라 후보별 좋아요 입력이다.
- 후보 목록 아래에 후보 간편 등록 웹을 여는 `점심 후보 보태기`와 사내 웹 리더보드를 여는 `전당 둘러보기` 링크 버튼을 둔다. 버튼은 강조 색상을 사용하지 않는다.
- 가격, 지도 링크, 추천인이 있을 때만 해당 정보를 표시한다.
- 선택값이 없을 때 `정보 없음` 같은 대체 문구나 빈 구분자를 표시하지 않는다.

## 채널 온보딩

- 채널을 열 때 안내 메시지를 한 번 게시하고 봇이 즉시 자동 고정한다.
- 안내에는 평일 오전 11시 게시, 복수 번호 반응 허용, 후보 추가 방법과 전당 확인 방법을 포함한다.
- 채널 북마크를 둘 경우 Google Spreadsheet는 `점심 후보`, 웹 리더보드는 `밥라투스트라의 전당`으로 구분한다.
- 자동 고정은 `pins:write`를 사용한다. 정기 추천 메시지는 고정하지 않는다.
- 채널 북마크 추가는 한 번뿐인 관리 작업이므로 `bookmarks:write`를 요청하지 않고 사람이 수행한다.

## 공정 순환 추천

복잡한 점수나 가중치를 사용하지 않고 추천 이력의 마지막 추천 시각만 이용한다.

1. 유효한 행을 식당 이름으로 묶는다.
2. 각 식당이 마지막으로 추천된 시각을 `recommendation_log`에서 찾는다.
3. 한 번도 추천되지 않은 식당을 가장 오래된 것으로 취급한다.
4. 마지막 추천 시각이 오래된 식당부터 최대 세 곳을 선택한다.
5. 마지막 추천 시각이 같으면 해당 식당들 사이에서 무작위로 순서를 정한다.
6. 선택된 식당에 메뉴가 여러 개라면 가장 오래 추천되지 않은 메뉴를 선택하고, 동률이면 무작위로 정한다.

식당과 메뉴를 비교할 때 앞뒤 공백과 연속된 공백만 정리한다. 별도의 내부 ID, 가중치, 점수, 추천 제외 기간은 도입하지 않는다.

식당명이 수정되면 과거 로그와 다른 식당으로 인식되어 우선 추천될 수 있다. 이는 구조를 단순하게 유지하기 위해 허용하는 한계이며, 직원은 같은 식당의 여러 메뉴 행에 같은 식당명을 사용한다.

## 같은 날짜의 중복 게시 방지

- 게시 전에 추천 로그를 읽어 현재 KST 날짜와 대상 Slack 채널 ID가 같은 행이 있는지 확인한다.
- 한 행이라도 있으면 해당 날짜와 채널의 게시가 완료된 것으로 보고 실행을 중단한다.
- 채널이 다르면 같은 날짜라도 별개의 실행으로 취급한다.
- 중복으로 종료하더라도 그 전에 최근 다섯 메시지의 좋아요 동기화는 수행한다.

## 게시와 로그 기록 순서

1. 후보와 기존 추천 로그를 읽고 검증한다.
2. 최근 다섯 추천 메시지의 좋아요를 조회해 기존 로그에 한 번에 반영한다.
3. 같은 KST 날짜와 채널의 기존 실행이 없는지 확인한다.
4. 잘못된 후보 행은 운영 채널에 한 번에 알리고 유효한 후보로 추천을 계속한다.
5. 추천을 계산하고 Slack 메시지를 게시한다.
6. 게시 성공 시 Slack 메시지 ID와 후보들을 추천 로그에 한 번에 추가한다.
7. 로그 기록까지 성공하면 후보 순서에 맞는 번호 반응을 추가한다.

로그 기록에 실패한 뒤 수동 재실행하면 기존 로그만으로 중복을 판별할 수 없으므로, 운영자가 Slack 채널의 실제 게시 여부를 먼저 확인해야 한다. 별도의 예약 상태나 실행 테이블은 초기 복잡성을 줄이기 위해 두지 않는다.

현재 메시지 형식은 다음과 같다.

```text
📜 밥라투스트라는 이렇게 말했다.

점심은 스스로 정해지지 않는다. 선택되어야 한다.
오늘 그대들 앞에는 세 갈래의 길이 놓여 있다.

1. 마루식당 — 제육볶음 · 10,000원 · 추천: 철수 <지도 링크|지도>
2. 소바집 — 냉소바 <지도 링크|지도>
3. 대성집 — 순댓국 · 10,000원 · 추천: 영희

마음이 가는 번호에 반응해주세요.
```

## 후보가 부족할 때

### 유효한 식당이 한두 곳인 경우

- 존재하는 후보만 게시한다.
- 후보가 세 곳보다 적다는 사실을 간결하게 알린다.
- 구성원이 Google Sheet에 새로운 후보를 추가하도록 안내한다.

후보가 두 곳이면 현재 형식은 다음과 같다.

```text
📜 밥라투스트라는 이렇게 말했다.

오늘 보이는 점심의 길은 둘뿐이다.

1. 마루식당 — 제육볶음
2. 소바집 — 냉소바

새로운 후보는 아래 버튼으로 보태주세요.
```

### 유효한 식당이 없는 경우

- 사용자용 점심 채널에는 메시지를 게시하지 않는다.
- 비공개 `#밥괘-운영` 채널에 운영 오류를 알린다.

## 운영 오류 알림

- 비공개 `#밥괘-운영` 채널을 사용한다.
- 봇과 실제 운영 담당자만 채널에 참여한다.
- Slack의 Channel Manager 역할을 API로 자동 탐색하여 수신자를 정하지 않는다.
- 사용자용 점심 채널에는 기술 오류를 노출하지 않는다.
- 같은 오류에 대해 운영 담당자 각각에게 중복 DM을 보내지 않는다.

다음 상황은 운영 채널 알림 대상이다.

- Google Sheet를 읽지 못해 추천을 만들 수 없음
- 잘못된 후보 행을 제외하고 유효한 후보로 추천을 계속함
- 최근 좋아요 동기화에 실패하여 기존 값을 유지함
- 유효한 추천 후보가 없음
- Slack 추천 메시지 전송이 최종적으로 실패함
- 메시지는 게시됐지만 추천 로그 기록에 실패함
- 메시지와 추천 로그는 완료됐지만 번호 반응 추가에 실패함

운영 메시지에는 인증 정보, 토큰 또는 불필요한 내부 데이터를 포함하지 않는다. 실패 시각, 실패 단계, 최종 결과와 추적 가능한 오류 식별자만 간결하게 표시한다.

좋아요 동기화 실패는 기존 집계값을 유지하고 당일 추천을 계속한다. 반면 번호 반응 추가 실패는 이미 메시지와 로그가 완료된 상태이므로 자동 재게시하지 않는다. 두 경우 모두 운영 채널과 systemd journal에서 확인한다.

## 재시도와 실행 제한

- Google Sheet 후보와 추천 로그 읽기는 실패하면 2초 간격으로 최대 세 번 시도한다.
- 최근 Slack 반응 조회와 좋아요 Sheet 갱신을 묶은 동기화도 실패하면 2초 간격으로 최대 세 번 시도한다. 같은 값을 다시 읽고 쓰므로 결과가 중복되지 않는다.
- Slack 추천 게시, 추천 로그 추가와 번호 반응 추가는 자동 재시도하지 않는다. 일부 단계만 성공한 상황에서 중복 게시나 불완전한 기록을 만들 수 있기 때문이다.
- 모든 시도가 끝나도 Sheet 읽기가 실패하면 게시하지 않고 운영 알림을 시도한다.
- 좋아요 동기화가 최종 실패하면 기존 값을 유지하고 당일 추천은 계속한다.
- systemd oneshot 작업은 120초 안에 끝나지 않으면 실패로 종료한다.
- systemd가 일일 작업 실패를 감지하면 별도 oneshot 작업이 운영 채널에 generic 알림을 보낸다. 애플리케이션이 이미 단계별 알림을 보낸 실패에서는 두 알림이 함께 보일 수 있다.
