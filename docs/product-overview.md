# 제품 개요

## 목표

사내 Slack에서 구성원들이 점심 선택에 드는 고민을 줄이고, 회사 구성원들이 알고 있는 좋은 식당과 메뉴를 함께 축적하고 활용할 수 있는 점심 추천 봇을 만든다.

현재 제품명은 **밥라투스트라**다. 이름과 사용자에게 보이는 표현은 [`brand.md`](brand.md)를 따른다.

## 제품 원칙

- Slack 채널에 참여한 구성원이 추천을 확인하는 옵트인 방식으로 운영한다.
- 초기 추천 데이터는 외부 장소 API가 아니라 사내 구성원이 공동 편집하는 Google Sheet에서 가져온다.
- 구성원이 직접 데이터를 추가하고 수정할 수 있도록 신뢰를 기반으로 운영하며 별도 승인 절차를 두지 않는다.
- 첫 버전은 단순하게 유지하되, 참고 사례의 기능을 완성 기준이나 제품의 상한선으로 삼지 않는다.
- 새로운 기능은 실제 사용 중 발견된 문제와 기대 효과를 바탕으로 선택한다.
- 외부 장소 API, 별도 관리 대시보드와 개인화 등은 필요성이 확인되면 이후 단계에서 검토한다.

## 현재 확정된 사항

- 현재 제품명은 `밥라투스트라`다.
- Slack 앱 형태로 구현한다.
- 추천을 게시할 전용 Slack 채널을 사용한다.
- Google Sheet를 초기 추천 데이터의 원본이자 공동 관리 화면으로 사용한다.
- 초기에는 개발자의 개인 Google 계정이 Sheet와 Cloud 프로젝트를 소유하고, 인턴 종료 전에 정직원 후임자에게 인계한다.
- 직원은 별도 승인 없이 시트의 추천 후보를 직접 추가하고 수정한다.
- 한 행은 하나의 `식당 + 메뉴` 추천 후보를 의미한다.
- 지도 링크는 선택값이며, 값이 있을 때만 Slack 메시지에 표시한다.
- 초기 단계에서는 외부 장소 API를 사용하지 않는다.
- 월요일부터 금요일까지 오전 11시(KST)에 점심 전용 채널로 자동 추천을 게시한다.
- 대한민국 공휴일을 별도로 판별하지 않으며, 공휴일이 평일이면 동일하게 게시한다.
- 한 번에 서로 다른 식당 세 곳을 추천한다.
- 자동 추천 메시지에서 `@channel` 또는 `@here` 멘션을 사용하지 않는다.
- 개인 명령형 추천은 초기 범위에 포함하지 않는다.
- 초기 애플리케이션은 Python 단발성 작업으로 구현하고 systemd timer로 실행한다.
- Slash Command나 이벤트를 받는 상시 Slack 서비스는 초기 범위에 포함하지 않는다.
- 추천은 식당별 카드로 나누지 않고 하나의 간결한 목록으로 표시한다.
- 공개 추천의 등장과 선언에 캐릭터성을 집중하고, 정보와 후속 안내는 간결하고 친절하게 쓴다.
- 브랜드 이모지는 절제해서 사용하고 제목의 `📜` 하나만 사용한다.
- em dash는 일반 문장에서 최소화하고 후보 행의 `식당 — 메뉴` 구분에는 사용할 수 있다.
- 가격, 지도 링크, 추천인이 없으면 빈 자리나 대체 문구 없이 해당 정보만 생략한다.
- 유효한 식당이 한두 곳이면 존재하는 후보만 게시하고 시트에 후보 추가를 안내한다.
- 유효한 식당이 없거나 운영 오류가 발생하면 사용자 채널에 기술 오류를 게시하지 않고 비공개 `#밥괘-운영` 채널로 알린다.
- 식당과 메뉴는 추천 이력을 이용한 공정 순환 방식으로 선택한다.
- 추천 이력은 같은 Google Spreadsheet의 봇 전용 `recommendation_log` 탭에 기록한다.
- 추천 로그가 명세와 다르면 공정 순환과 중복 방지를 위해 전체 실행을 중단한다.
- 같은 KST 날짜와 Slack 채널의 추천 로그가 있으면 중복 게시하지 않는다.
- Slack 게시 성공 후 추천 로그를 일괄 기록하며, 로그 기록 실패를 이유로 게시를 자동 재시도하지 않는다.
- Google Sheet 읽기와 최근 좋아요 동기화처럼 중복 게시를 만들지 않는 작업만 2초 간격으로 최대 세 번 시도한다.
- Slack 추천 게시, 추천 로그 추가와 번호 반응 추가는 자동 재시도하지 않는다.
- systemd 일일 작업이 실패하면 별도 oneshot 알림 작업이 비공개 운영 채널에 journal 확인을 안내한다.
- 각 후보에는 `1️⃣`, `2️⃣`, `3️⃣` Slack 반응을 대응시켜 구성원이 여러 후보에 좋아요를 표시할 수 있게 한다.
- 상시 이벤트 수신 없이 매일 최근 다섯 개의 밥라투스트라 메시지 반응 수를 추천 로그에 동기화한다.
- 최근 반응 동기화가 실패하면 기존 값을 유지하고 당일 추천은 계속한다.
- 좋아요 데이터는 처음부터 수집하되 자동 통계 게시 기능은 실제 사용을 확인한 뒤 결정한다.

Google Sheet의 정확한 스키마와 관리 규칙은 [`data/google-sheets.md`](data/google-sheets.md)를 따른다.
정기 게시와 예외 처리의 상세 정책은 [`features/daily-recommendation.md`](features/daily-recommendation.md)를 따른다.
번호 반응과 통계 데이터 수집 정책은 [`features/reactions.md`](features/reactions.md)를 따른다.
기술 스택과 실행 구조는 [`architecture/runtime.md`](architecture/runtime.md)를 따른다.

## 아직 결정하지 않은 사항

- 월간 밥라투스트라 결산을 게시할지와 구체적인 통계 항목
- 중복 시트 행을 직원에게 경고하거나 정리할 필요가 있는지

미결정 사항은 구현 전에 선택지, 장단점, 트레이드오프와 추천안을 제시하고 사용자와 합의한 뒤 확정한다.

## 향후 발전 방향

다음 항목은 확정된 로드맵이 아니라 실제 필요가 확인될 때 검토할 후보이다.

- 필요할 때 개인적으로 다시 추천받는 명령 또는 버튼
- 누적 반응을 추천 순위나 개인 선호에 반영
- 인원, 가격, 거리, 음식 종류 등의 조건 선택
- 휴무, 폐업 또는 부정확한 정보의 신고와 관리
- 외부 장소 API를 이용한 주소, 거리, 영업 정보 보완
- Google Sheet가 불편해질 경우 가벼운 관리 대시보드로 전환
- 구성원 또는 팀 단위의 개인화

후보라는 이유만으로 미리 구현하지 않는다. 각 기능은 실제 문제, 사용 빈도, 구현 및 운영 비용을 확인하고 선택한다.
