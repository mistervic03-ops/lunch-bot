# 프로젝트 문서 안내

이 디렉터리는 `AGENTS.md`에 담기에는 지나치게 상세하지만, 이후 작업자가 프로젝트를 이해하고 안전하게 변경하는 데 필요한 정보를 보관한다.

## 현재 문서 목록

- `README.md`: 상세 문서의 목차와 문서 관리 규칙을 설명한다.
- `product-overview.md`: 제품 목표, 현재 범위, 확정 사항과 미결정 사항을 설명한다.
- `brand.md`: 현재 제품명, 브랜드 콘셉트, 기본 문구와 말투 원칙을 설명한다.
- `architecture/runtime.md`: 기술 스택, 실행 모델, 컴포넌트 경계와 선택 이유를 설명한다.
- `data/google-sheets.md`: 공동 편집 Google Sheet의 데이터 구조와 관리 규칙을 설명한다.
- `features/daily-recommendation.md`: 정기 추천의 정상 동작, 메시지 구성과 예외 처리 정책을 설명한다.
- `features/reactions.md`: 후보별 번호 반응, 좋아요 집계와 향후 통계 활용 규칙을 설명한다.
- `integrations/slack.md`: Slack App Manifest, 최소 권한, 앱 설치, 채널 연결과 담당자 인계를 설명한다.
- `operations/development.md`: 현재 구현 상태, 코드 구조, 로컬 환경과 검증 및 systemd 템플릿 주의 사항을 설명한다.
- `operations/google-access.md`: Google Sheet와 Cloud 프로젝트의 조직 소유, 권한 및 담당자 인계 절차를 설명한다.
- `operations/sustainability.md`: 1년 이상 운영할 때의 데이터 규모, 외부 API 한도와 운영상 주의 사항을 설명한다.

정보가 실제로 생길 때 아래 기준에 따라 필요한 문서만 추가한다.

## 문서 추가 기준

다음과 같은 정보가 생기면 별도 문서로 분리한다.

- 시스템 구조와 주요 구성 요소의 관계
- 중요한 기술 선택과 그 배경 및 트레이드오프
- 기능별 동작, 제약, 예외 사항
- 개발 환경 설정, 테스트, 배포, 운영 절차
- 외부 서비스 연동 방법과 주의 사항
- 여러 작업에서 반복해서 확인해야 하는 도메인 지식

문서를 추가할 때는 다음을 지킨다.

1. 하나의 문서에는 명확한 목적을 부여한다.
2. 파일명과 제목만으로 다루는 내용을 예측할 수 있게 한다.
3. 사실, 결정 사항, 아직 확정되지 않은 내용을 구분한다.
4. 결정 기록에는 선택한 방법뿐 아니라 검토한 대안과 선택 이유도 남긴다.
5. 관련 구현이 바뀌면 문서도 같은 작업에서 갱신한다.
6. 이 문서의 목록과 루트 `AGENTS.md`의 문서 지도를 함께 갱신한다.

## 권장 확장 구조

아래 구조는 필요할 때만 만든다.

```text
docs/
├── README.md                     # 문서 목차와 관리 규칙
├── architecture/                 # 시스템 구조와 구성 요소
├── decisions/                    # 주요 의사결정과 트레이드오프
├── features/                     # 기능별 동작과 제약
└── operations/                   # 개발, 테스트, 배포, 운영 절차
```

프로젝트 성격에 맞지 않는 디렉터리는 만들지 않으며, 새로운 분류가 더 명확하다면 그 분류를 사용한다.

## 문서 갱신 체크리스트

작업 완료 전에 확인한다.

- 이번 변경으로 기존 설명이 틀리거나 불완전해지지 않았는가?
- 새로운 에이전트가 변경 이유와 동작을 문서만으로 찾을 수 있는가?
- 장기적으로 지켜야 할 새 규칙이 있다면 `AGENTS.md`에 반영했는가?
- 세부 구현이나 결정 사항을 적절한 문서에 기록했는가?
- 문서가 추가되거나 이동했다면 모든 문서 목록과 링크를 갱신했는가?
